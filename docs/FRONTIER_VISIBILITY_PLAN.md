# Frontier Visibility Plan

**Status:** Working execution plan  
**Purpose:** Make the right people at frontier labs, AI companies, research circles, and capital networks see `Nous` in the right order

---

## The Objective

We are not trying to create generic awareness.

We are trying to create **high-quality signal** around a specific claim:

> `Nous` is not "memory for LLMs." It is an attempt to define a new category: a persistent epistemic substrate for AI, with the language model acting as larynx rather than mind.

The plan therefore has to achieve three things in sequence:

1. make the category legible
2. make the system believable
3. make the project strategically interesting

This is not broad marketing.

It is staged attention design.

---

## The Audiences

We do not actually need "Google" or "Anthropic" as abstractions.

We need four audience types:

### 1. Research minds

People who care whether the category claim is intellectually real.

They react to:

- the Larynx Problem
- benchmark misalignment
- architectural inversion
- novelty with conceptual precision

### 2. Reliability / interpretability / safety people

People who feel the pain of fluent systems with weak epistemic integrity.

They react to:

- contradiction authority
- uncertainty boundaries
- calibration
- state and introspection

### 3. Agent / product / platform leads

People who can see a new layer in the AI stack.

They react to:

- "LLM is not the whole system"
- substrate as infrastructure
- leverage over existing models
- product surface that changes model behavior

### 4. Capital / corp dev / founder layer

People who think in strategic position, not only elegance.

They react to:

- category creation
- first credible artifact
- hard-to-copy architecture
- founder clarity

---

## The Three Public Objects

Before outreach, three things must exist in public form:

### A. Doctrine Object

A short, clear statement of the category claim.

Minimum materials:

- README hero
- `The Larynx Problem`
- `NOUS_STRATEGIC_DOCTRINE.md`

### B. Demo Object

A specific mechanism that makes the claim feel real.

Priority target:

- contradiction authority

Secondary targets:

- graph-prioritized steering
- gap-aware clarifying questions
- evidence crystallization

### C. Artifact Object

A place people can click, scan, and understand quickly.

Minimum materials:

- clean repo top
- canonical diagram
- coherent wiki path
- short video or live demo

## 30-Day Plan

### Week 1 — Make It Legible

Goal: anyone landing on the repo should understand what category claim is being made.

Deliverables:

- README hero rewritten around doctrine
- canonical `Industry vs Nous` diagram
- wiki reading path:
  - Home
  - The Larynx Problem
  - Benchmark
  - Architecture
- one-line and three-line summaries used consistently

Success condition:

- a new visitor can explain `Nous` in one minute without using the phrase "memory for LLMs"

### Week 2 — Make It Believable

Goal: show one mechanism that behaves like a different kind of system.

Deliverables:

- contradiction authority prototype
- one short public demo clip
- one atlas screenshot where the field feels like the center
- one example showing the graph correcting, blocking, or reshaping a model answer

Success condition:

- someone can say "this is not just retrieval"

### Week 3 — Make It Discussed

Goal: create public language around the category.

Deliverables:

- one canonical essay/thread:
  - "language models are the larynx, not the mind"
- one repo post with doctrine + demo + repo links
- one community launch path:
  - X / Hacker News / targeted research communities

Success condition:

- the project starts being described back to us in our own terms

### Week 4 — Make It Reach The Labs

Goal: place a small, high-quality signal packet in front of the right people.

Deliverables:

- one short doctrine packet
- one demo packet
- one target list for:
  - Anthropic
  - Google DeepMind
  - Mistral
  - OpenAI
  - independent researchers

Success condition:

- inbound interest, reply, or discussion from credible external people

---

## Priority Order

This order should not be reversed:

```text
doctrine
    -> demo
        -> artifact
            -> discussion
                -> outreach
```

If outreach happens before doctrine and demo are clear, we burn attention before we deserve it.

---

## The Signal Packet

Every external packet should be tiny.

### Structure

1. one sentence: what `Nous` is
2. one sentence: why the recipient should care
3. one doctrine link
4. one demo link
5. one repo link

### Example core sentence

`Nous` is an attempt to define a missing AI layer: a persistent epistemic substrate that sits behind the language model and gives it contradiction authority, knowledge boundaries, and longitudinal structure.

### Example why-you-should-care sentence

If your work touches reliability, agents, or model cognition, this may be relevant because it treats the language model as the larynx rather than the mind.

---

## What The Repo Must Never Default Back To

- "AI memory tool"
- "RAG but better"
- "benchmark winner"
- "multi-provider chat utility"

Those may describe parts of the system.

They must not become the identity of the system.

---

## Metrics For The Visibility Effort

We should measure whether the outside world is understanding the right thing.

Suggested metrics:

- how often external descriptions use `epistemic substrate`
- how often people paraphrase the larynx claim correctly
- inbound quality, not only raw quantity
- whether discussion centers on category, architecture, and mechanism rather than only benchmark score

---

## The One-Line Summary

> The attention plan for `Nous` is to make the category clear, the mechanism undeniable, and the strategic value legible — in that order.
