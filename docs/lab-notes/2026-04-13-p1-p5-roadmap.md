**From Contradiction Detection to Adaptive Self-Regulation**

## 1. The Roadmap Logic

The P1-P5 phases were designed as a dependency chain. Each phase builds on the previous, and skipping any phase breaks the feedback loop:

```
P1: Detect contradictions ──┐
                            ├──▶ Measurement
P2: Accumulate evidence ────┘
                            │
                            ▼
P3: Reflect on measurements ──▶ Self-adjustment
                            │
                            ▼
P4: Direct the LLM ────────────▶ Bottom-up control
                            │
                            ▼
P5: Track longitudinally ─────▶ Adaptive regulation
```

Without P1+P2, the system has nothing to measure. Without P3, measurement doesn't lead to change. Without P4, the substrate can't influence the LLM. Without P5, there's no way to tell if changes are improvements.

## 2. P1: Contradiction Detection

*Commit: 8e30315*

The system detects when new information contradicts existing knowledge. Each contradiction is flagged with:

- **Severity**: how strongly the new evidence conflicts with existing beliefs
- **Caught vs. acted on**: not all contradictions warrant action — some are noted but deferred
- **Catch rate**: the fraction of potential contradictions that are detected

The contradiction detector compares new relation evidence against existing edges. If a new edge with high evidence contradicts an existing high-evidence edge, the system flags it and optionally adjusts evidence scores.

Key metrics: `contradiction_caught`, `contradiction_acted_on`, `contradiction_catch_rate`, `contradiction_severity_mean`.

## 3. P2: Evidence Accumulation

*Commit: 8e30315*

Evidence accumulation provides three baseline metrics for the system's epistemic health:

- **Crystallization rate**: fraction of edges with weight > 0.55 — how much of the graph is "solid knowledge" vs. "tentative observation"
- **Evidence quality**: mean evidence_score on crystallized edges — how well-supported is the solid knowledge
- **Gap map shrink rate**: change in the number of nodes with high uncertainty — is the knowledge frontier expanding or contracting

These metrics are logged to `eval_log.jsonl` every NightRun cycle, creating the longitudinal dataset that P5's trend analysis operates on.

## 4. P3: Causal Reflection (Reflection-to-Policy Bridge)

*Commit: 435af2d*

P3 closes the loop between measurement and behavior. The `living_core` module performs reflection on the current state (energy, contradiction rates, crystallization) and generates observations. The `cognitive_policy` module evaluates these observations against trigger rules and adjusts system parameters:

| Trigger | Condition | Policy Change |
|---|---|---|
| energy < 0.3 | System depleted | extraction_threshold +0.10 (be more selective) |
| contradiction_catch_rate < 0.1 | Missing contradictions | evidence_floor -0.05 (review more) |
| bisociation_quality < 0.3 | Poor cross-domain links | lam_override +0.10 (more creative) |
| gap_map_shrink_rate < 0 | Knowledge frontier contracting | curiosity_priority = "high" |

The key insight: **reflection without policy change is observation.** P3 makes observation consequential — the system doesn't just notice that it's doing poorly, it adjusts its behavior in response.

## 5. P4: Substrate→LLM Direction

*Commit: c67641d*

P4 reverses the typical LLM→tool direction: the substrate tells the LLM what to focus on, not the other way around. Three mechanisms:

1. **Focus agenda**: The system prompt includes the current active drive and relevant goal context, directing the LLM's attention toward knowledge gaps
2. **Gap questions**: When the LLM responds, the substrate identifies concepts with low evidence and injects follow-up questions about them
3. **Hallucination block**: When the LLM generates claims about concepts with no supporting evidence in the substrate, the system flags or blocks the output

This is the architectural consequence of the Larynx Problem: if the LLM is the larynx (expression organ), then the substrate must provide the thought that precedes expression. P4 implements that direction.

## 6. P5: Evalving Harness + Operator Feedback

*Commits: 8e30315, b717b25*

P5 provides the longitudinal measurement backbone and human correction signal:

**Evalving harness:**
- `eval_log.jsonl` — one entry per NightRun cycle with all baseline metrics
- Trend analysis — detect rising/falling/stable patterns over 10-cycle windows
- Decline detection — if a metric falls 3 cycles in a row, generate a policy suggestion
- Auto-application — evalving policy suggestions flow through cognitive_policy

**Operator feedback (P5.3):**
- `feedback_log.jsonl` — thumbs up/down from the human operator
- Feedback summary — ratio of good/bad verdicts, recent trend
- Feedback signal — integrated into goal priority computation (0.20 weight)

The operator feedback loop is critical: it prevents the system from optimizing for metrics at the expense of human values. Without it, a system could achieve high crystallization by being very conservative (never adding new knowledge) — technically correct but practically useless.

## 7. How P1-P5 Set the Stage for the Drive Engine

The Intrinsic Drive Engine (D1–D6, documented separately) extends P1-P5 in a specific way:

- P1-P2 provide the **measurements** that the Drive Engine uses to generate goals (evidence gaps, contradictions, crystallization deficits)
- P3 provides the **policy adjustment** mechanism that goal metrics feed into
- P4 provides the **LLM direction** channel that goal-directed execution uses
- P5 provides the **longitudinal tracking** that makes goal satisfaction measurable over time

The Drive Engine adds the missing component: **persistent goals that direct autonomous behavior.** Without P1-P5, goals would have no basis for generation, no way to influence behavior, and no way to evaluate satisfaction. With P1-P5, the Drive Engine completes the architecture.
